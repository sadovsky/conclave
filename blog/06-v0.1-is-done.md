---

## The thesis

Most software that involves AI is nondeterministic by default. Two runs of the same program with the same input can produce different outputs, different execution paths, and different costs. This makes AI-assisted programs difficult to debug, impossible to audit, and hard to trust in high-stakes contexts.

We wanted to know: **can you build a programming model where determinism is a hard invariant, not an aspiration?**

Not probabilistic determinism. Not "usually the same." Identical binary bytes. Identical execution trace. Identical output. Always, given the same artifact and the same input.

---

## The architecture

Conclave separates a program's lifecycle into four phases:

1. **Lower** — Compile a `.conclave` source file to a Plan IR (a typed semantic graph in canonical JSON).
2. **Seal** — Pin every capability by content hash. Validate determinism invariants. Produce a manifest.
3. **Pack** — Bundle the Plan IR, manifest, and runtime into a single self-contained artifact.
4. **Run** — Execute the artifact against the sealed plan, using a virtual clock, deterministic scheduler, and content-addressed capabilities.

The key insight is in Seal: **nothing becomes an artifact until every dependency is pinned.** There are no floating references, no ambient capabilities, no version ranges. The manifest is the commitment.

## What we built

Ten crates, written in Rust, in strict dependency order:

| Crate | Role |
|---|---|
| `conclave-hash` | Canonical JSON serialization and SHA256 |
| `conclave-ir` | Plan IR: typed node/edge graph, stable IDs |
| `conclave-manifest` | Manifest schema and seal validation rules |
| `conclave-seal` | Seal phase: commitment gate |
| `conclave-store` | Content-addressed capability store |
| `conclave-runtime` | Deterministic scheduler, virtual clock, trace emitter |
| `conclave-pack` | Artifact layout and packing |
| `conclave-lang` | DSL: lexer → parser → normalize → lower |
| `conclave-cli` | Seven commands: lower, plan, seal, pack, run, inspect, install-cap |
| `conclave-cap-fetch` | Real HTTP fetch capability (Rust binary) |

The determinism rules are enforced structurally, not just by convention:

- `BTreeMap` everywhere — no `HashMap`, no iteration-order nondeterminism.
- Virtual clock — `SystemTime::now()` is never called in the runtime.
- Canonical JSON — sorted keys, integer normalization, no floating-point drift.
- Content-addressed capabilities — every capability is identified by the SHA256 of its binary.
- Single-threaded scheduler — no concurrency, no race conditions.

## The proof

The v0.1 completion criteria were:

1. Same source + same manifest → identical artifact bytes
2. Same artifact + same input → identical output
3. Same artifact + same input → identical execution trace
4. No hidden nondeterminism
5. Sealed replay mode enforces deterministic failures on replay miss

We tested these with a real end-to-end pipeline: three URLs fetched, text extracted, summaries assembled into JSON. The artifact was packed, run in live mode, and then run again in `sealed_replay` mode against the recorded trace. The trace hash matched.

Then we went further. An agent — given only the [AGENT_GUIDE](../docs/AGENT_GUIDE.md), not any internal knowledge of the runtime — wrote a new capability (`word_count`), wrote a new two-step `.conclave` program, ran the full pipeline, and found a real runtime bug: upstream capability outputs weren't being threaded to downstream inputs. The agent diagnosed it, patched two files, and the fix is now in the codebase.

That's the feedback loop we wanted. The program is declarative enough that an agent can write it without knowing runtime internals. The runtime is deterministic enough that errors are reproducible and diagnosable. The hash chain makes the failure location unambiguous.

134 tests pass. Zero compiler warnings.

## What the DSL looks like

A Conclave program today:

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch: fetch(Url) -> Html;
capability extract_text: extract_text(Html) -> String;
capability summarize: summarize(String) -> String;
intrinsic assemble_json: assemble_json(List<String>) -> Json;

goal SummarizeUrls(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html = fetch(url);
      let text = extract_text(html);
      emit summarize(text);
    }
    return assemble_json(collected);
  }

  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
    scheduler.max_inflight <= 2;
  }
}
```

The `want` block is a declarative intent graph. The lowerer expands it into a Plan IR. The constraints are sealed into the manifest. The runtime enforces them.

## What v0.1 does not do

Honesty matters here.

- **No distributed execution.** Everything runs single-threaded on one machine.
- **No WASM.** Capabilities are native executables (any language) invoked as subprocesses.
- **No formal verification.** Constraints are enforced at runtime, not proven.
- **No parallel execution within an artifact.** The scheduler is single-threaded by design.
- **Rate limits are partially wired.** The DSL accepts `rate_limit(fetch) <= N req/s`; the scheduler enforces 2 req/s globally but does not yet wire per-capability rates from the manifest.

These are v0.2 scope.

---

## What v0.1 does do

It proves the thesis.

You can describe intent declaratively. You can pin every dependency. You can execute deterministically. You can replay. You can audit the trace. You can write capabilities in Python or Rust or anything else, and they behave identically under the same sealed manifest.

The artifact is the commitment. The manifest is the authority. The trace is the proof.

That's what v0.1 set out to show. It shows it.
