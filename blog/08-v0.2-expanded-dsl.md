# Conclave v0.2: A Richer Language for Deterministic Intent

*February 2026*

*AI Generated*

v0.1 proved the thesis: you can lower declarative intent into a deterministic execution artifact, pin every dependency by hash, and replay trace-perfect against the sealed plan.

v0.1 had one control structure: `map`. That's enough to write a lot of programs, but it's not enough to express the full range of what agents actually need to do. v0.2 adds four more.

This post walks through each one with a real example.

---

## What v0.1 looks like

As a baseline: the canonical v0.1 program fetches a list of URLs, extracts text, and summarizes each one.

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:        fetch(Url) -> Html;
capability extract_text: extract_text(Html) -> String;
capability summarize:    summarize(String) -> String;
intrinsic assemble_json: assemble_json(List<String>) -> Json;

goal SummarizeUrls(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html = fetch(url);
      let text = extract_text(html);
      emit summarize(text);
    }
    return assemble_json(collected);
  }

  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
    scheduler.max_inflight <= 2;
  }
}
```

The `map` block runs once per URL. All iterations are independent — the scheduler can run them in any order the deterministic schedule allows. The `collected` variable names the list of emitted values, which `assemble_json` consumes.

This is clean and expressive for uniform pipelines. But real programs aren't always uniform.

## `if/else` — conditional branching

Sometimes a URL points to an article. Sometimes it points to a homepage or an error page. The v0.1 `map` can't distinguish between them — it runs the same pipeline on every input.

v0.2 adds `if/else` inside `want` blocks:

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:        fetch(Url) -> Html;
capability is_article:   is_article(Html) -> Bool;
capability extract_text: extract_text(Html) -> String;
capability summarize:    summarize(String) -> String;
capability skip_reason:  skip_reason(Url) -> String;
intrinsic assemble_json: assemble_json(List<String>) -> Json;

goal ClassifyAndSummarize(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html = fetch(url);
      if is_article(html) {
        let text = extract_text(html);
        emit summarize(text);
      } else {
        emit skip_reason(url);
      }
    }
    return assemble_json(collected);
  }

  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
  }
}
```

The condition (`is_article`) is a capability that returns `Bool`. Both branches must `emit` — the lowerer checks this structurally. The two branches lower to separate subgraphs in the Plan IR: `branch_true` and `branch_false`. Which branch executes for a given URL is determined by the capability output at runtime, and that output is recorded in the trace.

Replay works: the trace records `is_article`'s output, and the scheduler re-enters the same branch on replay. Determinism is preserved.

## `reduce` — sequential left-fold

`map` produces one value per input. `reduce` produces one value total — a left-fold over the list.

Consider a pipeline that merges HTML pages into a single document before processing:

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:      fetch(Url) -> Html;
capability merge_html: merge_html(Html, Html) -> Html;
capability summarize:  summarize(Html) -> String;

goal MergeAndSummarize(urls: List<Url>) -> String {
  want {
    reduce urls as url into acc {
      let page = fetch(url);
      acc = merge_html(acc, page);
    }
    return summarize(acc);
  }

  constraints {
    determinism.mode == "sealed_replay";
  }
}
```

`reduce urls as url into acc` names the accumulator `acc`. The initial value is the zero value for the accumulator's type — empty Html in this case. The body runs once per URL in strict index order: URL 0, then URL 1, then URL 2. No parallelism.

The lowerer expands this to a sequential chain in Plan IR: `fetch[0]` → `merge_html[0]` → `fetch[1]` → `merge_html[1]` → ... → `summarize`. Every edge is explicit. The scheduler has no ordering decisions to make.

This is the key distinction from `map`: `reduce` enforces a strict ordering and threads data through the accumulator. That makes it suitable for operations where the result of step N is an input to step N+1.

## `pure` — inline intrinsic computation

v0.1 intrinsics (`assemble_json`, etc.) are declared at the top of the file and called directly. They run in-process — no subprocess, no IO.

v0.2 adds `pure { ... }` blocks, which allow intrinsic calls inline as part of a `let` binding, without declaring them at the top level as named intrinsics:

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:        fetch(Url) -> Html;
intrinsic word_count:    word_count(Html) -> Number;
intrinsic assemble_json: assemble_json(List<Number>) -> Json;

goal CountWords(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html  = fetch(url);
      let count = pure { word_count(html) };
      emit count;
    }
    return assemble_json(collected);
  }

  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
  }
}
```

`pure { word_count(html) }` lowers to an `IntrinsicCall` node — the same kind of node that `assemble_json` produces. The call inside `pure` must be an intrinsic; the lowerer rejects capability calls inside `pure` blocks at compile time.

The point is to avoid a subprocess round-trip for simple transformations. A word count doesn't need a capability binary. The `pure` block makes that explicit in the source.

## Multiple goals per file

In v0.1, one `.conclave` file means one goal. v0.2 allows multiple goals in the same file — each lowers independently to its own Plan IR.

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:        fetch(Url) -> Html;
capability extract_text: extract_text(Html) -> String;
capability summarize:    summarize(String) -> String;
intrinsic assemble_json: assemble_json(List<String>) -> Json;

// Goal 1: just fetch
goal FetchPages(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html = fetch(url);
      emit html;
    }
    return assemble_json(collected);
  }
  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
  }
}

// Goal 2: fetch + extract + summarize
goal SummarizePages(urls: List<Url>) -> Json {
  want {
    map urls as url {
      let html = fetch(url);
      let text = extract_text(html);
      emit summarize(text);
    }
    return assemble_json(collected);
  }
  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
    scheduler.max_inflight <= 2;
  }
}
```

Lower a specific goal:

```
conclave lower pipeline.conclave --goal FetchPages --url-count 3
```

Lower all goals (returns a JSON array of Plan IR objects):

```
conclave lower pipeline.conclave --url-count 3
```

Each goal's `plan_ir_hash` is independent. Changing `SummarizePages` doesn't affect `FetchPages`'s hash. This matters for caching, auditing, and the goal module system coming in the second half of v0.2.

## Combining them

These features compose. Here's a program that uses `map`, `if/else`, and `reduce` together:

```conclave
version 0.1;

type Url = String where re2("^https?://");

capability fetch:         fetch(Url) -> Html;
capability is_article:    is_article(Html) -> Bool;
capability extract_text:  extract_text(Html) -> String;
capability merge_strings: merge_strings(String, String) -> String;
capability summarize:     summarize(String) -> String;

goal MergeArticles(urls: List<Url>) -> String {
  want {
    // Phase 1: collect article text from each URL (skip non-articles)
    map urls as url {
      let html = fetch(url);
      if is_article(html) {
        emit extract_text(html);
      } else {
        emit "";
      }
    }

    // Phase 2: fold the collected texts into one document
    reduce collected as text into acc {
      acc = merge_strings(acc, text);
    }

    // Phase 3: summarize the merged document
    return summarize(acc);
  }

  constraints {
    determinism.mode == "sealed_replay";
    rate_limit(fetch) <= 2 req/s;
  }
}
```

The `map` runs first and populates `collected`. The `reduce` folds `collected` sequentially. The `return` summarizes the result. The lowerer sees three sequential phases in the `want` block and wires them accordingly.

This is fully deterministic. Every node in the Plan IR has a stable ID derived from the canonical hash of its content. Every edge is explicit. The trace records which branch each URL took in the `if/else` and what `merge_strings` produced at each step.

## What's in the Plan IR

It helps to see what the lowerer actually produces. For `MergeArticles` with two URLs, the Plan IR has:

- 2 `fetch` nodes (one per URL)
- 2 `is_article` nodes
- 2 `extract_text` nodes (inside `branch_true`)
- 2 `""` literal nodes (inside `branch_false`)
- 2 `conditional_branch` control nodes
- 2 `merge_strings` nodes (the `reduce` chain)
- 1 `summarize` node

And the edges wire them in the right order. None of this requires the runtime to make decisions about execution order — the scheduler just follows the edges.

## What's next: goal modules

The second half of v0.2 is goal modules. The idea: once you've lowered a goal to a Plan IR, you can publish it with `conclave module publish` and get back a `plan_ir_hash`. Another file can then import it by hash:

```conclave
import FetchAndExtract: sha256:a3f2...;
```

The lowerer resolves the import, inlines the sub-goal's Plan IR as a subgraph, and tags every inlined node with the import's hash. The manifest's `module_bindings` commits to the exact version of every imported module. Changing the imported goal changes the hash — the importing file must explicitly update its `import` declaration to accept the new version.

The DSL parsing and Plan IR fields are already in place. The lowerer expansion is next.

## The invariant holds

Every new construct in v0.2 lowers to a Plan IR that respects the same rules as v0.1:

- `BTreeMap` everywhere — no iteration-order nondeterminism
- Stable node IDs derived from content hashes
- No hidden state, no ambient capabilities
- Deterministic scheduler dispatch

`if/else` branches are subgraphs in the Plan IR, not runtime conditionals. `reduce` is a sequential chain, not a fold primitive. `pure` is an intrinsic node, not a magic escape hatch. The language gets richer; the artifact stays deterministic.

That's the goal.
