# Conclave Conformance Tests (v0.1)

This document defines a small set of *golden* conformance cases for Conclave v0.1 implementations.

---

## 1. Deterministic scheduler trace â€” 3 URLs (rate-limited fetch)

### 1.1 Scenario

Goal: `SummarizeUrls([u1,u2,u3])`

Policy:

- `fetch_mode = replay`
- `rate_limit(fetch) = 2 req/s` (token bucket, 1s virtual windows)
- `max_inflight = 2`
- ready queue ordering key: `(url_index, node_kind, node_id)`
- node kind order: `FETCH < EXTRACT < SUMMARIZE < ASSEMBLE`
- tie-breaker: stable

Nodes:

- For each URL `ui`:
  - `Fi = fetch(ui)` (rate-limited)
  - `Ei = extract_text(Fi)` (local)
  - `Si = summarize(Ei)` (local)
- Final:
  - `A = assemble_json([R1,R2,R3])`

Assumed replay durations (deterministic from replay store):

- `F1`: 40ms
- `F2`: 70ms
- `F3`: 40ms

Local ops assumed (deterministic):

- `E*`: 15ms
- `S*`: 85ms
- `A`: 4ms

> Your implementation may measure different local durations; for deterministic conformance, either (a) treat local ops as 0ms in virtual time, or (b) use fixed modeled durations. The **event ordering** below is the primary conformance target.

### 1.2 Golden dispatch ordering

Expected DISPATCH order:

1. `F1`
2. `F2`
3. `E1`
4. `S1`
5. `E2`
6. `S2`
7. *(wait for next rate window)*
8. `F3`
9. `E3`
10. `S3`
11. `A`

### 1.4 Acceptance criteria

A conforming runtime MUST:

- produce the same DISPATCH ordering (Section 1.2)
- enforce the rate limit such that `F3` is not dispatched before the next window boundary
- output a deterministic error if replay data is missing (see Section 2)

## 2. Replay miss produces deterministic error

Given the same scenario but the replay store lacks `u2` response:

Expected:

- Node `F2` fails with `ERR_REPLAY_MISS`
- Error object includes:
  - `code = ERR_REPLAY_MISS`
  - `node_id` (or symbolic node ref) for `F2`
  - details contain the normalized URL
